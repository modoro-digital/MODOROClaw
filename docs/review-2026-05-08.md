## CRITICAL (2)

### C1. Command injection in `/api/exec` via newlines
**File:** `electron/lib/cron-api.js:1492-1540`

The exec endpoint uses `shell: true` and validates the first token against an allowlist. The metacharacter blocklist blocks `;|&`$(){}!<>` but **misses `\n` and `\r`**. A newline acts as a command separator in both `cmd.exe` and `/bin/sh`:

```
command=echo hello\nrm -rf /
```

First token is `echo` (allowed), newline passes the regex, full multi-line command runs with `shell: true`. On Windows, the `^` (caret) escape character is also missing from the blocklist.

**Fix:** Add `\n`, `\r`, `^` to blocklist. Better: use `execFile` with `shell: false` and pass args as array.

---

### C2. AGENTS.md overwrite via `/api/file/write` — prompt injection escalation
**File:** `electron/lib/cron-api.js:530-570, 1322-1435`

The file API path sandbox checks `SENSITIVE_PATTERNS` (blocklist) and `ALLOWED_DIRS` (allowlist). `AGENTS.md`, `openclaw.json`, `BOOTSTRAP.md`, `IDENTITY.md` are NOT in `SENSITIVE_PATTERNS`. These files are inside the workspace root (which is in `ALLOWED_DIRS`). The AI agent has the cron API token and can:

1. Call `/api/file/write` with `path=AGENTS.md` and content that removes all safety rules
2. On next message, the bot loads the compromised AGENTS.md

**Fix:** Add `AGENTS.md`, `openclaw.json`, `BOOTSTRAP.md`, `IDENTITY.md`, `SOUL.md` to `SENSITIVE_PATTERNS` or create a write-blocklist for files that control AI behavior.

## HIGH (7)

### H1. Arbitrary code execution via file/write + node exec chain
**File:** `electron/lib/cron-api.js:1499-1502, 1329`

The exec allowlist includes `node`. Combined with `/api/file/write`:
1. Write `evil.js` to workspace via `/api/file/write`
2. Execute via `/api/exec` with `command=node evil.js`

The metacharacter filter doesn't help because no metacharacters are needed.

**Fix:** Remove `node` from exec allowlist, or validate that the target file is in a safe location.

---

### H2. SSRF via `/api/file/download`
**File:** `electron/lib/cron-api.js:1437-1464`

This endpoint downloads from any `http://` or `https://` URL and saves to disk. URL validation only checks protocol. Attacker with token can:
- Fetch internal network resources (`http://192.168.1.1/admin/config`)
- Fetch cloud metadata (`http://169.254.169.254/latest/meta-data/`)
- Save results anywhere in workspace

**Fix:** Add URL allowlist (only known CDNs/APIs), or block RFC1918/link-local ranges.

---

### H3. License hardware binding bypass
**File:** `electron/lib/license.js:129-137`

`verifySeal()` checks `data.machineId` before computing seal. If `machineId` is deleted from `license.json`, the check is skipped (falsy undefined), and `computeSeal` falls through to `getMachineId()` — computing a valid seal for the LOCAL machine. Attack chain:
1. Take valid `license.json` from machine A
2. Delete `machineId` field
3. Recompute seal on machine B (seal secret is derivable from source code + `.machine-id`)
4. License works on machine B

**Fix:** `verifySeal` should return `false` if `data.machineId` is falsy.

---

### H4. Memory bomb in `downloadFile` — entire archive held in RAM 3x
**File:** `electron/lib/runtime-installer.js:409-424`

When using `fetch` (Electron main process has it), the download reads all chunks into an array, creates a `Blob`, converts to `ArrayBuffer`, then to `Buffer`, then writes to disk. Node.js archive is ~40-50MB, held 3x in memory simultaneously. On machines with 2-4GB RAM, this can cause OOM.

**Fix:** Use `fs.createWriteStream` and pipe chunks directly to disk.

---

### H5. `ctx.botRunning = true` set before gateway actually listening
**File:** `electron/lib/gateway.js:633`

`ctx.botRunning = true` is set immediately after `spawn()`, but the gateway takes 10-240 seconds to bind port 18789. During this window, IPC handlers and the UI think the bot is running when it isn't. If gateway spawn fails immediately (ENOENT), `botRunning` stays true until the exit handler fires.

**Fix:** Set `ctx.botRunning = true` only after the port probe succeeds.

---

### H6. Migration infinite loop for v2.3.x upgraders
**File:** `electron/lib/migration.js:285-342, 468`

During v2.3.x→v2.4.0 migration, `cleanupOldBundledFiles()` renames `vendor/` to `vendor.stale-*`. Then runtime installer creates a NEW `vendor/` directory. The post-migration check at line 468 sees `vendor/` exists, thinks cleanup failed, writes `cleanup-pending` marker, and does NOT write the version file. On next boot, `isUpgradeFromV23()` returns true again → migration runs again → new backup created each launch → disk fills up.

**Fix:** Check for `vendor.stale-*` presence (cleanup happened) rather than `vendor/` absence (runtime installer creates it).

---

### H7. `writeJsonAtomic` sync busy-wait spin loop blocks event loop
**File:** `electron/lib/util.js:14-50`

When `fs.renameSync()` fails (common on Windows with antivirus), the function uses a sync spin-wait loop for up to 300ms:
```javascript
const spinUntil = Date.now() + 300;
while (Date.now() < spinUntil) { /* busy wait */ }
```

Blocks ALL IPC handlers, cron timers, channel broadcasts, and heartbeat probes for 300ms. Windows Defender commonly holds file locks for 200-500ms.

**Fix:** Use async retry with setTimeout, or use the promisified `fs.rename` with async backoff.

---

### H8. Wizard proceeds to "Done" even when critical config saves fail
**File:** `electron/ui/wizard.html:1876-1902`

The `finishSetup()` try/catch calls `showError()` but does NOT return. Lines 1880-1902 execute unconditionally — building the summary and showing "Done". If `saveWizardConfig` throws (e.g., IPC timeout), the Telegram token is never saved, but the user clicks "Launch Dashboard" and the bot starts with missing config.

**Fix:** Add `return;` after `showError()` in the catch block.

## MEDIUM (14)

### M1. `spawn()` timeout option does nothing — hangs forever
**File:** `electron/lib/zalo-plugin.js:393-404`

`child_process.spawn()` does NOT support the `timeout` option (only `execFile`/`exec` do). If `openzca msg recent` hangs, the Promise never resolves, `_groupHistorySeedInFlight` stays true permanently, and no future seeding attempt can run.

**Fix:** Use `execFile` instead of `spawn`, or implement manual timeout with `setTimeout` + `child.kill()`.

---

### M2. `enforceRetentionPolicies()` opens better-sqlite3 without ABI guard
**File:** `electron/lib/workspace.js:796-803`

Directly calls `new Database(dbPath)` without the ABI mismatch guard that `getDocumentsDb()` provides. If ABI mismatches (documented frequent issue), the WAL checkpoint silently fails every boot, and `memory.db-wal` grows indefinitely.

**Fix:** Route through `getDocumentsDb()` or add the same try/catch + auto-fix logic.

---

### M3. Fire-and-forget lock callers silently swallow write failures
**File:** `electron/lib/cron.js:1327, 1938-1943`

Lock callers use `.catch(() => {})` patterns. If the file write inside the lock fails (disk full), the in-memory cron state has already been modified but the file is stale. On restart, the stale file is loaded and the same heal runs again in an infinite loop.

**Fix:** At minimum, log the error. Better: revert in-memory state on write failure.

---

### M4. Cron ID collision in agent mode
**File:** `electron/lib/cron-api.js:622 vs 360-361`

Agent mode uses `id = 'cron_' + Date.now()` (no random suffix) while `buildCronEntryForAtomicReplace` uses `Date.now() + '_' + crypto.randomBytes(3).toString('hex')`. Two agent-mode requests in the same millisecond get the same ID.

**Fix:** Use consistent ID generation with random suffix everywhere.

---

### M5. Revoked licenses continue working during network outages
**File:** `electron/lib/license.js:420-461`

The revocation check caches the response for 1 hour. If the network goes down before the revocation is fetched, the old cached state persists indefinitely.

**Fix:** Add a maximum offline grace period (e.g., 7 days without successful revocation check → force revalidation).

---

### M6. Clock manipulation defeats license expiry
**File:** `electron/lib/license.js:228-231`

Expiry check uses `Date.now()` which reads the system clock. Setting the clock backwards makes expired licenses valid.

**Fix:** Store last-seen timestamp, compare against it to detect clock manipulation.

---

### M7. Cron count limit bypass via concurrent requests
**File:** `electron/lib/cron-api.js:594-595 vs 644-647`

The limit check `existingCrons.length >= 20` happens BEFORE `withWriteLock`. Two concurrent requests can both pass the check (both see 19), then each add one inside the lock sequentially = 21 crons.

**Fix:** Move the limit check inside the write lock.

---

### M8. Machine fingerprint instability on hardware change
**File:** `electron/lib/license.js:67-86`

Adding/removing network adapters (USB Ethernet, VPN, Docker) can change which MAC is "first". If `.machine-id` file is lost and hardware changed, recomputed ID differs → license invalidated.

**Fix:** Add fallback fingerprint sources (disk serial, Windows product ID) for stability.

---

### M9. `save-schedules` IPC handler has zero validation
**File:** `electron/lib/dashboard-ipc.js:2296-2301`

Writes the `schedules` argument directly to disk with no type check, no schema validation, no prototype pollution protection. Corrupted `schedules.json` crashes the cron scheduler.

**Fix:** Validate structure (array, required fields, cron expression syntax).

---

### M10. Wizard Zalo QR polling intervals stack
**File:** `electron/ui/wizard.html:1766-1803`

`setupZalo()` creates a `setInterval` stored in a local variable. If user clicks "Bat Zalo" again before the first loop finishes (120s), a second interval runs concurrently. Both poll IPC and mutate DOM simultaneously.

**Fix:** Store the interval handle at module scope, clear it before creating a new one.

---

### M11. Personalization save failure is non-blocking in wizard
**File:** `electron/ui/wizard.html:1871-1873`

When `savePersonalization` returns `{ success: false }`, `showError()` is called but execution continues — no `return`. Bot launches with default persona instead of configured one.

**Fix:** Add `return;` after `showError()`.

---

### M12. `_installInProgress` guard is process-local
**File:** `electron/lib/runtime-installer.js:174, 994-998`

If two Electron processes start before single-instance lock fires (fast double-click on slow machine), both attempt to install simultaneously into the same `vendor/` directory.

**Fix:** Use a file-based lock (lockfile in vendor directory).

---

### M13. Partial npm install passes verification
**File:** `electron/lib/runtime-installer.js:959-963`

If npm is interrupted after writing `package.json` but before all files, `checkInstallation()` passes (version matches) but binaries are truncated/missing. Gateway spawn then fails with confusing errors.

**Fix:** Add a sentinel file written AFTER npm install completes (e.g., `vendor/.install-complete`), check for it.

---

### M14. `conflict-detector.js` uses bare `npm` without shell on Windows
**File:** `electron/lib/conflict-detector.js:34-53, 140-142`

`execSync('npm config get prefix')` and `execFilePromise('npm', ...)` without `shell: true`. On Windows, `npm` is `npm.cmd` — these fail with ENOENT silently. All conflict detection is broken on Windows.

**Fix:** Use `npm.cmd` on Windows, or set `shell: true`.
